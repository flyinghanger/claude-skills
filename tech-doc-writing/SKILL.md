# 技术文档写作规范

基于"自顶向下的结构化表达 + 适度排版"原则，帮助用户写出清晰、易读的技术文档。

原文出处：[如何写好一篇文档?](https://bytetech.info/articles/7337928955418116115)

## 使用方法

```bash
/doc-writing <file.md> draft     # 基于用户提供的素材/要点，按规范生成文档初稿
/doc-writing <file.md> review    # 审查现有文档，按四阶段逐项给出改进建议
/doc-writing <file.md> rewrite   # 根据 review 结果直接重写/优化文档
```

## 核心理念

> **自顶向下的结构化表达 + 一点点的排版 = 一篇不错的文档**

好文档的标准：
- **作者视角**：让读者快速、清晰理解作者要表达的内容
- **读者视角**：读者能快速、清晰地了解到自己想要了解的内容

时间分配参考：定主题 10% → 立框架 20% → 填内容 60% → 调排版 10%

## 四阶段规范

### 一、定主题

**核心原则：主题是什么不重要，重要的是存在主题且忠于主题。**

- 文档的所有内容都应为主题服务
- 偏离主题的讨论应适当简化，一笔带过或附加链接
- 写作过程中反复回看主题，防止跑偏

**常见错误**：描述"如何从北京到上海"的方案，却花大量篇幅讲飞机的组成和高铁的制造。

### 二、立框架

**核心原则：框架的重点在于思考逻辑，而不是表达细节。**

框架的作用：
- 帮助梳理思路，防止遗漏要点
- 确保内容不偏题，抓住重点
- 抽象地看框架，更容易理清条理
- 读者更容易理解作者的思考逻辑

**框架分层**：

| 框架 | 载体 | 作用 |
|------|------|------|
| 主框架 | 目录 | 统领全文的行文逻辑 |
| 次框架 | 架构图/流程图 | 统领技术方案的描述逻辑 |

**技术方案标准模板**：

```
- 背景
- 目标
- 前期调研
  - 数据统计
  - 问题分类
  - 流程现状
- 技术方案
  - 概览
  - 具体方案
  - 数据指标
- 里程碑
- 相关文档
```

**关键**：下一节所讲的内容要符合读者的心理预期，不让人感觉跳脱。不必追求逻辑精妙，越简单的逻辑越容易理解。

### 三、填内容

**核心原则：结构化不应只体现在框架上，而是方方面面。**

#### 常见要素

**背景**：
- 必须站在没有相关知识背景的读者角度去描述
- 作者和读者之间通常存在显著信息差，作者需有意识地减少这种信息差
- 精简、直观的背景描述和案例列举，是带动读者"入局"的关键
- 常见错误：作者觉得方案理所应当，背景一两句话概括

**目标**：
- 显式强调目标，不要让读者去猜、去感受
- 目标应是事先定好的，而非总结而来（自顶向下）
- 只有明确目标，读者才能评估后续内容的合理性

**方案**：
- 先画架构图梳理框架，再填内容
- 架构图的价值：人对图的接受程度远大于大段文字；图的理解粒度可粗可细；提升文档的专业度

#### 实用技巧

1. **先由少到多，再由繁至简**：先顺思路写，再根据主题和框架精简内容
2. **读者视角逐句通读**：抛开脑补，以读者身份一字一句通读，发现冗余和逻辑不通
3. **避免口语化**：使用书面用语，精简表达
4. **一图胜千言**：当需要大段文字描述逻辑关系时，优先用图表达
5. **善用表格**：分类/分模块描述一系列事情时，表格是最佳工具

### 四、调排版

**核心原则：排版应是不减分的状态。读起来规整、有序、高效，不让人感觉不适。**

#### 四大设计原则

**对齐**：
- 画图尤其注意多个元素间的对齐关系
- 善用参考线和对齐操作（左右对齐、均匀分布）
- 慎用首行缩进，用空行明确段落
- 段落在读者常用屏幕下尽量不超过 5 行

**对比**：
- 对比要强烈，不是略有不同而是截然不同
- 善用分栏做左右对比
- 不同元素间也可以对比（如标题加 emoji/序号强化与正文的区分）

**重复**：
- 标题样式、列表符号、缩进、间距等保持一致
- 重复隐含"这些是存在某种关联的一系列对象"的暗示
- 带来专业性和权威性

**亲密性**：
- 刻意将关系更近的内容放在一起（分组）
- 画图时尤为重要，通过分层体现模块关系

> 设计排版往往不是单一使用某个原则，而是综合使用多种原则。

## 执行流程

### draft 模式

1. 读取用户提供的素材/要点文件
2. **定主题**：从素材中提炼核心主题，确认与用户对齐
3. **立框架**：基于技术方案模板生成目录结构
4. **填内容**：按框架逐节填充，遵循背景→目标→方案的叙事逻辑
5. **调排版**：应用四大设计原则，结构化呈现内容
6. 输出完整文档

### review 模式

按四个维度逐项审查，输出改进建议表格：

| 维度 | 检查项 | 当前状态 | 改进建议 |
|------|--------|----------|----------|
| 主题 | 是否存在明确主题 | | |
| 主题 | 内容是否忠于主题 | | |
| 框架 | 目录逻辑是否自顶向下 | | |
| 框架 | 各节顺序是否符合读者心理预期 | | |
| 内容 | 背景是否充分（读者视角） | | |
| 内容 | 目标是否显式声明 | | |
| 内容 | 是否存在口语化表达 | | |
| 内容 | 大段文字是否可用图/表替代 | | |
| 排版 | 对齐是否一致 | | |
| 排版 | 段落是否过长（>5行） | | |
| 排版 | 重复元素是否一致 | | |
| 排版 | 相关内容是否分组（亲密性） | | |

### rewrite 模式

1. 先执行 review 流程
2. 根据 review 结果逐项修改
3. 以读者视角通读一遍，确认逻辑通顺
4. 输出修改后的文档，附带改动说明
