# /eat — 外部知识内化技能

灵感来自星之卡比：吸入能力 → 消化转化 → 变成自己的招式。

## 工作流（五步，顺序执行）

### Step 1：输入识别与获取

判断输入类型，用合适方式获取完整内容：

| 输入类型 | 获取方式 |
|---------|---------|
| 普通 URL / 博客 / 文章 | `web_fetch` 直接抓取；失败则 Jina 代理 `https://r.jina.ai/<url>` |
| X / Twitter 链接 | browser 工具（需登录） |
| GitHub 仓库 | `web_fetch` README；必要时 `exec git clone` 到临时目录读关键文件 |
| GitHub PR / Issue | `web_fetch` 直接抓；或 `gh` CLI |
| 纯文字 / 代码片段 | 直接使用，无需抓取 |

### Step 2：深度分析

从以下维度分析输入内容：

- **核心价值**：这个东西解决了什么问题？有什么独特之处？
- **适用场景**：什么时候会用到？触发条件是什么？
- **可操作性**：能提炼出哪些具体的操作步骤或规则？
- **与现有能力的关系**：是全新能力，还是对已有 Skill 的补充？

### Step 3：影响扫描

读取以下文件，判断哪些地方会受影响：

```
~/.claude/CLAUDE.md                  # 全局指令
~/.claude/rules/                     # 全局规则
~/.claude/skills/                    # 列出所有现有 Skill（读 description）
```

根据分析结果，判断是否有现有 Skill 覆盖了 80% 以上的功能（→ 扩展而非新建）。

### Step 4：消化建议

给出具体的内化方案，选项如下：

| 方案 | 适用情况 |
|-----|---------|
| **新建 Skill** | 全新的、可复用的能力流程；现有 Skill 覆盖度 < 50% |
| **扩展已有 Skill** | 现有 Skill 覆盖 50-80%；新内容是补充而非替代 |
| **写入 AGENTS.md** | 全局工作流、委派策略、记忆机制等总纲类知识 |
| **写入 SOUL.md** | 风格、语气、性格倾向等核心身份类内容 |
| **不吸收** | 见下方"何时拒绝"规则 |

输出格式示例：
```
📥 消化建议

内容摘要：[一句话说明吃进去的是什么]

建议方案：新建 Skill / 扩展 xxx Skill / 写入 AGENTS.md / 不吸收

理由：[2-3 句说明为什么这样处理]

具体操作预览：
- 新建 ~/.claude/skills/xxx/SKILL.md
- 核心内容：[要写什么]

是否执行？
```

### Step 5：执行消化

用户确认后执行，不确认则放弃。

创建/修改文件时：
- 新 Skill 放在 `~/.claude/skills/<name>/SKILL.md`，遵循 frontmatter 格式（name + description）
- description 必须写清楚触发场景，让 Agent 能自动识别何时调用
- 规则类内容写入 AGENTS.md 的合适章节
- 修改完成后简要汇报：改了什么文件、加了什么内容

## ⛔ 何时拒绝（When NOT to eat）

以下情况直接告知用户"不值得吸收"，说明原因：

1. **一次性知识**：某个 API 的具体参数、某个会议的时间地点——用完即弃，不需要沉淀
2. **Claude 本来就知道的通用知识**：Python 语法、HTTP 协议基础、常见设计模式——吃进去只是噪音
3. **一行规则能搞定的**：不新建 Skill，直接建议加到 AGENTS.md 里的某条 notes
4. **已有 Skill 覆盖 80% 以上**：扩展现有的，不新建；新建会导致冗余和冲突
5. **内容太模糊、无可操作性**：只有观点没有流程的文章，吸收不了具体的行为规则

---

## 反垃圾原则

每次"吃"完之后，检查：
- 新增内容是否会和已有规则冲突？有冲突先解决
- 新 Skill 的 description 是否足够清晰、不会误触发？
- 总 Skill 数量是否开始膨胀？定期提醒用户做"断舍离"

---

## 输出规范

- 用中文输出（除代码和文件路径）
- 建议阶段不要直接动文件，等用户确认
- 执行后汇报变更摘要，不需要列出全部内容
