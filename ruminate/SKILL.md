# /ruminate — 反刍与沉淀

像反刍动物一样，把已经「吃过」的工作经验再嚼一遍，提炼可复用的模式，沉淀为 Skill、Rule 或 Hook。

与 `/eat` 的区别：`/eat` 消化外部输入，`/ruminate` 消化自身工作经验。

## 工作流（四步）

### Step 1：回顾会话上下文

扫描当前会话，识别以下信号：

| 信号类型 | 具体表现 |
|---------|---------|
| **重复模式** | 同一类操作做了 2+ 次（同类搜索、同类文件处理、同类命令序列） |
| **临时 workaround** | 遇到问题后绕过的方式，可能值得固化为规则 |
| **决策点** | 做出的非显然选择（选了 A 而不是 B，为什么？） |
| **新发现** | 工作中发现的工具用法、API 特性、项目约定 |
| **用户纠正** | 用户说「不要这样」「应该那样」——最高优先级的沉淀信号 |
| **流程创新** | 会话中自然形成的新工作流（多步骤组合、工具链） |

### Step 2：分类评估

对每个识别到的信号，判断沉淀类型：

| 沉淀物 | 适用场景 | 存放位置 |
|--------|---------|---------|
| **Skill** | 可复用的多步骤流程，有明确的触发条件和输出 | `~/.claude/skills/<name>/SKILL.md` |
| **Rule** | 行为约束、决策规则、路由逻辑 | `~/.claude/rules/<name>.md` |
| **Hook** | 需要在特定事件自动触发的检测/提醒 | `~/.claude/hooks/` + `settings.json` |
| **Memory** | 用户偏好、项目上下文、非显然的经验 | `~/.claude/projects/*/memory/` |
| **改进现有** | 现有 Skill/Rule 覆盖了场景但缺少本次发现的细节 | 修改对应文件 |
| **不沉淀** | 一次性操作、已有覆盖、太模糊无法操作化 | 跳过 |

### Step 3：生成建议

扫描现有 Skill 和 Rule，判断重叠度后输出建议：

```
反思报告

会话摘要：[一句话概括本次会话做了什么]

---

发现 N 条可沉淀的模式：

1. [模式名称]
   - 来源：[会话中的具体事件]
   - 类型：新建 Skill / 新建 Rule / 改进 xxx / 写入 Memory
   - 理由：[为什么值得沉淀，解决什么问题]
   - 预览：[要写什么内容，一句话]

2. ...

---

不沉淀的项（简要说明原因）：
- [xxx]：一次性操作，不会复用
- [yyy]：已被 xxx Rule 覆盖

---

执行哪些？（输入编号，如 1,3，或 all / none）
```

### Step 4：执行

用户选择后执行对应操作：

- **新建 Skill**：创建目录 + SKILL.md，遵循 frontmatter 规范（name, description, version）
- **新建 Rule**：创建 `rules/<name>.md`，设置合适的 `paths` 范围
- **新建 Hook**：创建脚本 + 更新 `settings.json`
- **改进现有**：Edit 对应文件，说明改了什么
- **写入 Memory**：按 memory 规范写入对应文件

执行完毕后 git commit。

## 判断标准

### 什么值得沉淀

- 会话中**第二次**遇到的模式（第一次可能是偶然，第二次说明是规律）
- 用户的**纠正和偏好**（最高价值——避免未来重复犯错）
- 花了 **>3 步才搞定**的流程（可能值得封装成 Skill）
- 会话中**自然形成的决策树**（值得固化为 Rule）

### 什么不值得沉淀

- 项目特定的一次性操作
- 已经在 CLAUDE.md 或现有 Rule 中覆盖的行为
- 太模糊、无法写成可执行指令的「感悟」
- 代码层面的模式（应该体现在代码本身，不是 Rule）

## 注意事项

- 宁缺毋滥：3 条高质量沉淀 > 10 条低质量噪音
- 改进优先于新建：先看现有 Skill/Rule 能否扩展
- 每条建议必须可操作：不接受「以后注意 xxx」这种模糊建议
- 用户纠正 > 自己发现：用户明确说的偏好永远优先沉淀
