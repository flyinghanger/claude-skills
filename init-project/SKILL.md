# /init-project — 长周期项目初始化

为需要多次会话完成的项目创建结构化工作框架，让后续每次会话都能快速恢复上下文、按优先级推进。

灵感来源：Anthropic《Effective Harnesses for Long-Running Agents》中的 Initializer Agent 模式。

## 输入

用户提供以下任意形式的项目需求：
- PRD / 需求文档（URL 或文件路径）
- README 或项目说明
- 口头描述

如果当前目录已存在代码，先读取项目结构理解现状。

## 前置检查

如果项目根目录已存在 `features.json`，**不覆盖**，提示用户：
- 是否要重新初始化（覆盖现有文件）
- 还是继续使用现有框架（此时不需要 init-project，直接开始工作）

## 工作流（六步，顺序执行）

### Step 1：需求分析

1. 读取用户输入，理解项目目标和范围
2. 如已有代码，扫描项目结构（`ls`、读关键文件如 `package.json`、`go.mod`、`requirements.txt` 等）
3. 识别技术栈（语言、框架、构建工具、测试框架）
4. 向用户确认理解是否正确，有歧义时先对齐

### Step 2：生成 features.json

在项目根目录创建 `features.json`，格式：

```json
{
  "project": "项目名称",
  "created": "YYYY-MM-DD",
  "features": [
    {
      "id": 1,
      "title": "特性简称",
      "description": "一句话描述用户可感知的行为",
      "priority": "high",
      "dependencies": [],
      "test_criteria": [
        "可验证的测试条件 1",
        "可验证的测试条件 2"
      ],
      "passes": false
    }
  ]
}
```

原则：
- 每个 feature 必须有可验证的 `test_criteria`（不是模糊描述，是具体的用户操作 + 预期结果）
- 按依赖关系排序，被依赖的排前面
- `dependencies` 用 id 数组引用（如 `[1, 3]` 表示依赖 feature 1 和 3）
- 粒度适中：一个 feature 应能在一次会话内完成
- 所有 feature 初始 `passes: false`
- 小项目 5-10 个 feature，大项目 15-30 个
- **先展示给用户确认，再写入文件**

### Step 3：生成 claude-progress.md

在项目根目录创建 `claude-progress.md`：

```markdown
# 项目进度

## 项目概览
- **项目名称**：xxx
- **技术栈**：xxx
- **初始化时间**：YYYY-MM-DD
- **总特性数**：N

## 当前状态
- 已完成：0/N 个特性
- 进行中：无
- 下一步：Feature #1 - xxx

## 会话日志

### Session 1 — YYYY-MM-DD（初始化）
- 创建项目框架文件
- 分解为 N 个特性
- 下一步：Feature #1 - xxx
```

### Step 4：生成 init.sh

在项目根目录创建 `init.sh`，根据识别到的技术栈生成具体命令：

```bash
#!/usr/bin/env bash
set -euo pipefail

# Project: xxx
# Generated by /init-project

# Install dependencies
# （根据技术栈生成，如 npm install / pip install -r requirements.txt / go mod download）

# Start dev server (background)
# （根据技术栈生成，如 npm run dev & / python manage.py runserver & ）

# Verify environment
# （根据技术栈生成基础验证命令）

echo "Environment ready."
```

要求：
- `chmod +x init.sh`
- 脚本必须**幂等**（重复运行不会出错）
- 只包含环境准备命令，不包含业务逻辑

### Step 5：创建项目级 CLAUDE.md（可选）

如果项目有特殊的工作方式，在 `~/.claude/projects/<project-path>/CLAUDE.md` 创建项目级指令。

`<project-path>` 格式：将 `$PWD` 中的 `/` 替换为 `-`，去掉开头的 `-`。
例：`/Users/alice/myapp` → `Users-alice-myapp`

典型内容：
- 项目特有的构建/测试命令
- 特殊的代码规范或架构约束
- 关键技术决策说明

仅在项目确实有特殊需求时创建，不要为了创建而创建。

### Step 6：Git Commit

```bash
git add features.json claude-progress.md init.sh
git commit -m "Initialize long-running project structure

Generate features.json (N features), claude-progress.md,
and init.sh for multi-session development workflow."
```

## 注意事项

- 不要一次性生成代码，这个 Skill 只负责**框架搭建**，不写业务代码
- features 的拆分是关键——太粗会导致单次会话做不完，太细会增加管理开销
- test_criteria 要具体到「用户做什么 → 看到什么」，避免「功能正常工作」这类模糊描述
- 禁止后续 Coding Agent 删除或修改 features.json 中的 test_criteria，只允许改 `passes` 字段
